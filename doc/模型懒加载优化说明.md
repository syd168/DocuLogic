# 模型懒加载优化方案

## 问题描述

原项目在启动时（`startup_event`）就会立即加载模型，导致：
- 启动时即占用大量显存（可能超过10GB）
- 如果不需要解析文档，显存被白白占用
- 影响其他需要显存的程序运行

## 解决方案

实现**懒加载（Lazy Loading）**机制，模型只在真正需要时才加载：

### 核心改动

#### 1. 移除启动时的模型加载

**文件**: `web/app/main.py`

```python
@app.on_event("startup")
async def startup_event():
    """应用启动时不加载模型，改为懒加载模式"""
    # ... 其他初始化代码 ...
    
    print("\n🚀 应用启动完成！")
    print("💡 提示：模型将在首次上传文档时自动加载（懒加载模式）")
    print("💡 提示：模型空闲 5 分钟后将自动卸载以释放显存")
```

#### 2. 添加懒加载函数

新增三个核心函数：

- **`ensure_model_loaded()`**: 确保模型已加载，未加载时自动加载
- **`get_inference_model()`**: 获取模型实例，支持自动卸载
- **`unload_model()`**: 手动卸载模型释放显存

关键特性：
- ✅ 线程安全的懒加载（使用锁机制）
- ✅ 自动超时卸载（默认5分钟空闲后自动释放显存）
- ✅ 双重检查锁定（避免并发请求重复加载）

#### 3. 修改上传接口

**文件**: `web/app/main.py` - `/upload` 接口

```python
@app.post("/upload")
async def upload_document(...):
    # 懒加载：首次使用时才加载模型
    try:
        local_model = ensure_model_loaded()
    except Exception as e:
        raise HTTPException(status_code=503, detail=f"模型加载失败...")
    
    # 使用 local_model 进行推理
    result = await asyncio.to_thread(
        local_model.process_document, ...
    )
```

#### 4. 更新状态查询接口

所有检查模型状态的接口都改用 `get_inference_model()`：

```python
# 之前
"model_loaded": model is not None

# 现在
"model_loaded": get_inference_model() is not None
```

### 工作流程

```
用户访问系统
    ↓
应用启动（不加载模型，显存占用极小）
    ↓
用户上传文档
    ↓
触发 ensure_model_loaded()
    ↓
检查模型是否已加载
    ├─ 已加载 → 直接使用
    └─ 未加载 → 加锁加载模型
                ↓
           记录使用时间
                ↓
           返回模型实例
    ↓
执行文档解析
    ↓
更新最后使用时间
    ↓
等待下一次请求...
    ↓
如果5分钟内无新请求
    ↓
自动卸载模型，释放显存
```

### 配置参数

可以在 `main.py` 中调整超时时间：

```python
_model_unload_timeout: int = 300  # 模型空闲超时时间（秒），默认5分钟
```

### 优势

1. **节省显存**: 启动时不占用显存，仅在需要时加载
2. **自动管理**: 空闲时自动卸载，无需手动干预
3. **线程安全**: 多用户并发请求时只加载一次
4. **向后兼容**: 原有的模型重载接口仍然可用
5. **用户体验**: 对用户透明，无需改变使用方式

### 使用场景对比

| 场景 | 优化前 | 优化后 |
|------|--------|--------|
| 仅浏览系统 | 占用 ~12GB 显存 | 占用 < 1GB 显存 |
| 偶尔解析文档 | 持续占用 ~12GB | 用时加载，闲时释放 |
| 频繁解析文档 | 持续占用 ~12GB | 保持加载，5分钟超时 |
| 多用户并发 | 持续占用 ~12GB | 智能加载/卸载 |

### 注意事项

1. **首次加载延迟**: 第一次上传文档时需要加载模型，会有几秒到几十秒的延迟（取决于硬件）
2. **超时时间**: 可根据实际使用频率调整 `_model_unload_timeout`
3. **监控日志**: 关注日志中的模型加载/卸载信息，了解使用情况

### 相关接口

- `GET /api/settings`: 查看模型加载状态
- `POST /api/admin/model/reload`: 手动重新加载模型
- `GET /api/admin/model/status`: 查看模型+下载详细状态（支持按 engine 查询）
- `GET /api/admin/converter/{engine_id}/download/schema`: 获取插件下载能力
- `POST /api/admin/converter/{engine_id}/download/start`: 启动插件下载任务
- `GET /api/admin/converter/{engine_id}/download/status/{task_id}`: 查询插件下载任务状态
- `POST /api/admin/model/download`: 兼容旧接口（内部转发）

### 技术细节

#### 线程安全

使用 `threading.Lock` 确保并发安全：

```python
with _model_lock:
    if model is not None:
        return model
    
    # 加载模型...
```

#### 显存管理

卸载时执行完整的清理流程：

```python
del model
model = None
gc.collect()  # 垃圾回收
torch.cuda.empty_cache()  # 清理 CUDA 缓存
```

#### 状态跟踪

使用 `_last_used_time` 跟踪最后使用时间：

```python
if model is not None and _last_used_time is not None:
    idle_time = time.time() - _last_used_time
    if idle_time > _model_unload_timeout:
        unload_model()
```

## 测试建议

1. **启动测试**: 启动服务，观察初始显存占用
2. **首次上传**: 上传第一个文档，观察模型加载过程
3. **连续上传**: 短时间内多次上传，验证不会重复加载
4. **超时测试**: 等待5分钟，验证模型自动卸载
5. **并发测试**: 多用户同时上传，验证线程安全

## 回滚方案

如果需要恢复原来的行为（启动时加载），可以：

1. 在 `startup_event` 中添加 `ensure_model_loaded()` 调用
2. 设置环境变量 `EAGER_LOAD_MODEL=1`（需额外实现）

## 总结

通过实现懒加载机制，成功解决了启动时占用大量显存的问题，同时保持了系统的易用性和性能。这是一个向后兼容的优化，不会影响现有功能。
