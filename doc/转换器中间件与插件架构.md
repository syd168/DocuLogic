# DocuLogic 转换器中间件与插件架构（设计细化）

本文档在既定构思上进一步细化：**中间件（编排与契约）**、**与后端统一的协议（DTO）**、**插件（引擎实现与配置适配）** 三层边界，便于后续在 `converts/` 落地实现时对照，减少与 `web/` 主干的耦合与返工。

---

## 一、背景与目标

### 1.1 现状简述

当前文档解析主链路大致为：FastAPI 接收上传与任务状态，后台任务调用 `web/app/model_inference.py` 中的模型封装，再与 `converts/` 侧转换能力协同。引擎能力扩展（例如增加 PaddleOCR、其他版面或 OCR 管线）若继续堆在单类或单函数中，会出现：

- 主干分支膨胀、测试困难；
- 各引擎依赖与运行环境不同，难以隔离；
- 系统设置与任务参数难以按引擎扩展而不污染全局配置。

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| 插件化 | 新引擎以「插件」形式增加，尽量**少改** `web/` 与任务编排入口。 |
| 契约稳定 | FastAPI 仍只处理「上传 → 任务 → 进度 → 统一结果」，不因引擎数量分叉业务语义。 |
| 配置可扩展 | 各引擎专有参数通过 **schema + 默认值** 声明，由中间件校验与注入，避免 `AppSettings` 无限膨胀。 |
| 可演进 | 先进程内插件；必要时再引入进程外 Worker / 消息队列，而不推翻协议。 |

## 二、总体分层（概念对齐）

```text
┌─────────────────────────────────────────────────────────────┐
│  web（FastAPI）：上传、ParseJob、WebSocket、下载、鉴权       │
│  只依赖「统一任务协议」，不依赖具体引擎实现                    │
└───────────────────────────┬─────────────────────────────────┘
                            │ 调用（同步或异步封装均可）
┌───────────────────────────▼─────────────────────────────────┐
│  converts/ 中间件（Host）：路由、生命周期、契约校验、进度聚合   │
│  不写 Paddle/Qwen 细节，只调度已注册插件                       │
└───────────────────────────┬─────────────────────────────────┘
                            │ 插件接口（同一套 ABI）
┌───────────────────────────▼─────────────────────────────────┐
│  插件（Plugin / Adapter）：具体推理、引擎私有配置、资源管理     │
│  例：logics_parsing 插件、paddle_ocr 插件…                   │
└─────────────────────────────────────────────────────────────┘
```

**一句话归纳**：中间件 = 稳定边界 + 路由与生命周期；插件 = 引擎实现 + 引擎配置适配；后端 = 仍只和「统一结果」打交道。

## 三、第一层：中间件（`converts/` 内）

### 3.1 职责（必须做）

- **注册表**：维护 `engine_id → 插件实现` 的映射（启动扫描或显式注册）。
- **路由**：根据任务或系统策略选择插件（见 5.2）；支持 fallback 策略时在文档中写清优先级。
- **生命周期**：在调用插件前后做统一步骤，例如：
  - 工作目录与 `job_id` 隔离检查；
  - 取消信号（`threading.Event` 等）在页级/阶段级透传；
  - 超时与资源清理的钩子（由中间件计时，插件响应 `cancel`）。
- **契约校验**：对「统一输入 DTO」做必填校验；对插件声明的 **config schema** 做校验（见第五节）。
- **进度聚合**：将插件回调转换为 **统一进度事件**（见 4.3），再交给 `web` 已有 WebSocket 推送逻辑。

### 3.2 非职责（刻意不做）

- 不实现某一种模型的 `forward`、OCR 解码等；
- 不在此层写死与单一厂商路径绑死的逻辑（路径应来自配置或插件元数据）。

### 3.3 建议目录形态（实现时可调整）

```text
converts/
  middleware/                 # 或 engine_host/，名称以仓库约定为准
    __init__.py
    registry.py               # 插件发现与注册
    router.py                   # engine_id / 策略选择
    pipeline.py                 # 可选：链式步骤（预处理→推理→后处理）
    contracts/                  # DTO 与错误码常量
      job_input.py
      job_result.py
      progress.py
      errors.py
  plugins/
    logics_parsing_v2/
      plugin.py                 # 实现统一插件接口
      manifest.yaml             # 可选：engine_id、版本、能力标签
      config.schema.json        # 可选：JSON Schema
    paddle_ocr/
      ...
```

说明：**中间件代码放在 `converts/`** 与「转换能力同仓」一致，便于与现有转换器目录并列演进；`web` 仅通过**稳定 Python API**（例如 `run_conversion_job(job_input) -> job_result`）调用，避免 `web` 直接 `import` 各插件散点。

## 四、第二层：与 FastAPI 的统一协议（DTO / 契约）

本节是方案里**最关键、也最难一次设计好**的部分：后端与中间件之间只传递**结构化数据**，所有引擎差异在插件内消化，对外只剩同一套字段语义。

### 4.1 统一输入（建议字段，实现时可增删但需版本化）

| 字段 | 类型（概念） | 说明 |
|------|----------------|------|
| `protocol_version` | `str` | 契约版本，例如 `"1.0"`，便于以后平滑升级。 |
| `job_id` | `str` | 与现有 ParseJob、输出目录一致。 |
| `user_id` | `int \| null` | 鉴权与配额（若需要）。 |
| `engine_id` | `str` | 如 `logics_parsing_v2`、`paddle_ocr`；可由系统默认 + 任务覆盖。 |
| `input` | `object` | `path`（绝对或工作区内相对）、`mime` / `extension`、可选 `original_filename`。 |
| `output` | `object` | `work_dir`、与现有产物命名约定（如 `{job_id}.md`）对齐。 |
| `limits` | `object` | 如 `max_pdf_pages`、超时秒数、是否允许多文件等（与现有设置对齐）。 |
| `flags` | `object` | 如 `show_page_numbers`、`image_output_mode` 等**跨引擎通用**开关。 |
| `engine_config` | `object` | **仅本任务**引擎参数快照（已由中间件按 schema 校验）；来源可以是 DB 子集 + 任务级覆盖。 |
| `cancel_token` | 可序列化句柄或引用 | 进程内可为 `threading.Event` 包装；跨进程时可改为取消标记 ID，由插件轮询。 |

**原则**：FastAPI 只组装「业务已知的通用字段」+ `engine_id` + `engine_config`；**不向 FastAPI 暴露**各插件私有类型。

### 4.2 统一输出（与当前 `process_document` 返回值对齐思路）

| 字段 | 类型（概念） | 说明 |
|------|----------------|------|
| `status` | `str` | `completed` \| `failed` \| `stopped` \| `partial`（与现有任务状态可映射）。 |
| `output_files` | `list[str]` | 相对 `work_dir` 或绝对路径，与下载接口约定一致。 |
| `pages_parsed` | `int \| null` | 与现有 ParseJob 字段对齐。 |
| `user_stopped` | `bool` | 与现有取消语义一致。 |
| `partial` | `bool` | 部分完成。 |
| `error` | `object \| null` | 见 4.4 错误对象。 |
| `metrics` | `object \| null` | 可选：耗时、引擎版本、token 数等，便于运维。 |

### 4.3 统一进度事件（供 WebSocket 映射）

建议**事件信封**结构固定，**载荷**可扩展：

| 字段 | 说明 |
|------|------|
| `type` | `progress` \| `log` \| `phase_change`（可选）。 |
| `progress` | `int`，0–100，与现有前端进度条一致。 |
| `message` | 人类可读短文案。 |
| `phase` | 可选：`uploaded` / `routed` / `infer_page` / `postprocess` / `pack`。 |
| `engine_id` | 可选：便于前端展示「当前引擎」。 |
| `page` | 可选：当前页码，多页 PDF 时有用。 |

**约定**：插件只通过中间件提供的 `emit_progress(envelope)` 上报；**禁止**插件直接依赖 `web` 的 WebSocket manager，避免反向耦合。

### 4.4 错误模型与错误码（建议分级）

**错误对象建议字段**：

| 字段 | 说明 |
|------|------|
| `code` | 机器可读，稳定枚举或分层命名（见下表）。 |
| `message` | 给用户看的简短说明（可国际化 key）。 |
| `detail` | 仅服务端日志；默认不返回给前端或受环境变量控制。 |

**错误码示例（命名空间可按项目前缀调整）**：

| `code` | HTTP / 任务语义 | 说明 |
|--------|-------------------|------|
| `CONV_INVALID_INPUT` | 400 / failed | 输入路径不存在、格式不支持。 |
| `CONV_ENGINE_NOT_FOUND` | 400 / failed | `engine_id` 未注册。 |
| `CONV_CONFIG_INVALID` | 400 / failed | `engine_config` 未通过 schema。 |
| `CONV_CANCELLED` | — / stopped | 用户取消。 |
| `CONV_TIMEOUT` | — / failed | 超过任务或引擎级超时。 |
| `CONV_OOM` | — / failed | 显存/内存不足（插件可检测后映射）。 |
| `CONV_INTERNAL` | 500 / failed | 未分类异常，detail 入日志。 |

实现时可将 `code` 固化为常量模块，便于前后端与文档同步。

### 4.5 与现有 `web` 的衔接方式（落地提示）

- **短期**：在 `model_inference.LogicsParsingModel.process_document` 内部改为调用 `converts.middleware.run_job(...)`，由中间件再分发给 `logics_parsing_v2` 插件；对外函数签名与返回值结构**保持不变**，风险最低。
- **中期**：将「选 `engine_id` + 组装 `engine_config`」上移到 `process_document_task`，从 `AppSettings`/用户表读取配置块，再传入中间件。

## 五、第三层：插件（引擎 + 配置适配）

### 5.1 插件最小接口（概念 API）

每个插件应实现（名称可改为抽象基类或 Protocol）：

| 方法 | 职责 |
|------|------|
| `metadata()` | 返回 `engine_id`、人类可读名称、版本、`capabilities`（支持扩展名、是否多页、是否需要 GPU 等）。 |
| `config_schema()` | 返回 JSON Schema（或引用 `config.schema.json` 路径）；用于中间件校验与文档生成。 |
| `default_config()` | 返回默认 `engine_config` 对象。 |
| `validate_job(job_input)` | 可选：在路由后、执行前做最后一道校验（例如 Paddle 仅接受图像类输入）。 |
| `run(job_input, emit_progress) -> job_result` | 唯一重逻辑入口；内部再调具体推理。 |

### 5.2 路由与配置来源

| 来源 | 用途 |
|------|------|
| 系统默认 `engine_id` | 存在 `AppSettings` 或独立「引擎路由表」配置中，字段尽量**短且稳定**。 |
| 每用户默认（可选） | 用户表或扩展 JSON 列，优先级低于任务级。 |
| 单次任务覆盖（可选） | 上传接口 `Form` 字段或管理端「试跑」接口；中间件合并后做 schema 校验。 |

**与 `AppSettings` 的关系**：全局表只保留「路由 + 各引擎配置的**存储块**」（例如 `engine_configs JSON` 按 `engine_id` 分 key），**不要把每个 Paddle 参数都加成一列**，避免「一锅粥」。

### 5.3 `engine_config` 与 schema 工作流

1. 插件在注册时提交 **JSON Schema** 与 **default**。
2. 管理员在系统设置中编辑 JSON（或未来生成表单 UI）→ 保存到 DB。
3. 任务开始时，中间件 `merge(default, db_stored, job_override)` → `validate(schema)` → 注入 `job_input.engine_config`。
4. 插件 `run` 只读 `job_input.engine_config`，不再直接读全局单例。

### 5.4 能力声明（`capabilities`）建议字段

便于中间件与前端做能力探测：

| 字段 | 说明 |
|------|------|
| `input_extensions` | 如 `[".pdf", ".png"]`。 |
| `max_pages` | 是否支持多页、上限语义（与全局上限的关系）。 |
| `requires_gpu` | 布尔或 `preferred`。 |
| `output_modes` | 如 `["base64","separate","none"]` 与现有前端选项对齐。 |

## 六、端到端时序（与现有产品对齐）

以下与当前「上传 → BackgroundTasks → WebSocket」产品形态一致，仅把「模型调用」替换为「中间件 + 插件」。

1. 客户端 `POST /upload`，服务端落盘、建 `ParseJob`、`job_id`。
2. `process_document_task` 构造 **统一输入 DTO**（含 `engine_id`、`engine_config`）。
3. 调用 `converts` 中间件 `run_job`：路由 → 校验 → 插件 `run`。
4. 插件通过 `emit_progress` 上报 → 中间件转成 **统一进度事件** → 现有 WebSocket 推送。
5. 插件返回 **统一输出 DTO** → 任务更新 DB、completion 消息与下载列表与现网一致。

## 七、分阶段落地建议

| 阶段 | 内容 | 价值 |
|------|------|------|
| **P0** | 抽出 `contracts` + 注册表 + 单一 `logics_parsing_v2` 插件；`web` 仅改一处调用。 | 验证契约与目录结构，行为与线上一致。 |
| **P1** | 增加第二插件（如 PaddleOCR），路由规则与 `engine_config` 入库。 | 证明「增插件、少动主干」。 |
| **P2** | 可选：子进程 / 容器执行重引擎，中间件通过 IPC 或队列收结果；协议字段不变。 | 隔离依赖与 GPU 争用。 |

## 八、风险与约束（设计时一并考虑）

- **依赖冲突**：不同插件 Python/CUDA 版本不同 → P2 进程隔离优先于同一解释器硬塞。
- **超时与取消**：多页 PDF 须在页循环中检查取消；超时应映射为 `CONV_TIMEOUT` 并清理半写文件。
- **路径安全**：中间件统一校验 `work_dir` 与 `input.path` 不得逃逸出任务目录。
- **契约演进**：任何 DTO 字段变更必须 bump `protocol_version`，并在 README/本文档登记兼容策略。
